# Proton Mail Bridge Docker

[![Build and push](https://github.com/asyncura/ProtonMailBridgeDocker/actions/workflows/build-and-push.yml/badge.svg)](https://github.com/asyncura/ProtonMailBridgeDocker/actions/workflows/build-and-push.yml)

An always-up-to-date Docker image of the [Proton Mail Bridge](https://proton.me/mail/bridge) command-line interface. It runs a local SMTP/IMAP server inside a container so that other containers (or hosts on your network) can send and receive email through your Proton account.

![Logo Proton Mail Bridge docker](https://raw.githubusercontent.com/asyncura/ProtonMailBridgeDocker/master/logo.png "Merci à Korben pour le logo!")
_(Merci [Korben](https://korben.info/) pour le logo)_

> **Warning:** Proton Mail Bridge requires a **paid** Proton plan (Mail Plus, Proton Unlimited or Proton for Business). It does not work with a free account.

## Images

Images are rebuilt automatically whenever Proton releases a new Bridge version, for `linux/amd64` and `linux/arm64`, and published to both GitHub Container Registry and Docker Hub:

| Flavour | Base image | Tags |
|---|---|---|
| Debian (default) | `debian:trixie-slim` | `latest`, `debian`, `<version>`, `<version>-debian` |
| Alpine (smaller) | `alpine:3` | `alpine`, `<version>-alpine` |

```bash
docker pull ghcr.io/asyncura/proton-mail-bridge:latest   # or :alpine
docker pull asyncura/proton-mail-bridge:latest           # Docker Hub mirror
```

Both flavours share the same entrypoint and configuration, and ship a built-in container healthcheck.

## Quick start

### With docker compose (recommended)

```bash
# The bundled compose file expects an external network for DNS resolution
# between your containers; create it once:
docker network create --subnet 172.20.0.0/16 network20

wget https://raw.githubusercontent.com/asyncura/ProtonMailBridgeDocker/master/compose.yaml
docker compose up -d
```

For the Alpine flavour, use [compose.alpine.yaml](https://raw.githubusercontent.com/asyncura/ProtonMailBridgeDocker/master/compose.alpine.yaml) instead.

### With docker run

You **must** mount a volume on `/root` — it stores the Bridge configuration and your encrypted credentials, so they survive container updates:

```bash
mkdir -p /path/to/your/volume/storage

docker run -d --name=protonmail_bridge \
  -v /path/to/your/volume/storage:/root \
  -p 127.0.0.1:12025:25/tcp \
  -p 127.0.0.1:12143:143/tcp \
  --restart=unless-stopped \
  ghcr.io/asyncura/proton-mail-bridge:latest
```

This exposes SMTP on `127.0.0.1:12025` and IMAP on `127.0.0.1:12143`. Add `--network network20` if you created the shared network above.

Check that the container is up and that the Bridge started (you should see a bridge in ASCII art):

```bash
docker ps
docker container logs protonmail_bridge    # or: docker compose logs
```

## First-time setup: log in to your Proton account

The Bridge starts automatically with the container, but on first run it has no account yet. Open a shell in the running container and use the interactive Bridge CLI:

```bash
docker exec -it protonmail_bridge /bin/bash
```

Only one Bridge instance can run at a time, so stop the one started by the container before launching the interactive CLI:

```console
root@8972584f86d4:/app# pkill bridge
root@8972584f86d4:/app# /app/bridge --cli
....
      Welcome to Proton Mail Bridge interactive shell
....
>>> login
Username: test_account@proton.me
Password:
Authenticating ...
Two factor code: 123456
Account test_account was added successfully.
>>> A sync has begun for test_account.
Sync (test_account): 1.0% (Elapsed: 0.5s, ETA: 46.0s)
...
A sync has finished for test_account.
```

Tip: `Ctrl+Shift+V` pastes in most Linux terminals, and `help` lists all commands (see also the [Bridge CLI guide](https://proton.me/support/bridge-cli-guide)).

**If you use multiple addresses or custom domains**, switch to split address mode so each address gets its own credentials. This triggers another full sync:

```console
>>> change mode 0
Are you sure you want to change the mode for account test_account to split? yes/no: yes
Address mode for account test_account changed to split
```

### Get your SMTP/IMAP credentials

Run `info` and **copy the username and password** — the password is generated by the Bridge and is *not* your Proton account password:

```console
>>> info
Configuration for test_account@proton.me
IMAP Settings
Address:   127.0.0.1
IMAP port: 1143
Username:  test_account@proton.me
Password:  abcedfGHI12345
Security:  STARTTLS

SMTP Settings
Address:   127.0.0.1
SMTP port: 1025
Username:  test_account@proton.me
Password:  abcedfGHI12345
Security:  STARTTLS

>>> exit
root@8972584f86d4:/app# exit
```

Note: your clients must connect to the **published container ports** (12025/12143 in the examples above), **not** the internal ports shown by `info`.

Finally, restart the container so the Bridge runs under the container's supervision again (we killed it earlier):

```bash
docker container restart protonmail_bridge   # or: docker compose restart
```

The logs should end with `A sync has finished for name_of_your_account`.

## Configuration

All settings are environment variables, overridable at `docker run` / compose level:

| Variable | Default | Purpose |
|---|---|---|
| `CONTAINER_SMTP_PORT` | `25` | SMTP port the container listens on. Change only on a port conflict (e.g. another MTA on the same Docker network). |
| `CONTAINER_IMAP_PORT` | `143` | IMAP port the container listens on. |
| `PROTON_BRIDGE_SMTP_PORT` | `1025` | Internal port the Bridge listens on. On some Kubernetes/k3s setups the Bridge picks `1026` instead — verify with `netstat -ltpn` inside the container. |
| `PROTON_BRIDGE_IMAP_PORT` | `1143` | Internal IMAP port of the Bridge (may be `1144` on k3s, see above). |
| `PROTON_BRIDGE_HOST` | `127.0.0.1` | Internal address the Bridge listens on. |

The Bridge only listens on `127.0.0.1` inside the container; the entrypoint forwards `0.0.0.0:${CONTAINER_SMTP_PORT}` and `0.0.0.0:${CONTAINER_IMAP_PORT}` to it with `socat`.

### Notes

* Your email client may complain about the self-signed certificate used by the Bridge — this is expected.
* If other containers should only be able to *send* email, publish only the SMTP port and skip IMAP.
* The image includes a `HEALTHCHECK` that probes the SMTP port every 60 s, so `docker ps` shows the container as `healthy`/`unhealthy`.

## TrueNAS Scale

### TrueNAS Scale 24.10+ (apps as Docker containers)

Tested on TrueNAS Scale 24.10 following the [custom app installation](https://www.truenas.com/docs/truenasapps/usingcustomapp/) documentation. Create a [dataset](https://www.truenas.com/docs/scale/24.10/scaleuireference/datasets/) for the Bridge data before installing the app.

Recommended parameters:

1. **Image Configuration** — Image repository: `ghcr.io/asyncura/proton-mail-bridge` / Image tag: `latest` / Pull policy: `Always pull...`
2. **Container Configuration** — Entrypoint: `/app/entrypoint.sh` / Restart Policy: `Unless stopped`
3. **Network Configuration** — Add > Container Port: `25` / Host Port: `12025` (or any free port) / Protocol: `TCP`
4. **Storage Configuration** — Storage Add > Type: `Host path` / Mount Path: `/root` / Host path: `/mnt/path/to/your/protonmail-dataset`
5. **Resource Configuration** — Check `Enable resource limits` and configure to your liking.

### TrueNAS Scale 24.04 and older (apps as Kubernetes)

Tested following the [custom app installation](https://www.truenas.com/docs/scale/scaleuireference/apps/installcustomappscreens/) documentation.

Recommended parameters:

1. **Container images** — Image repository: `ghcr.io/asyncura/proton-mail-bridge` / Image tag: `latest` / Image pull policy: `Always pull...`
2. **Container Entrypoint** — Command: `/app/entrypoint.sh`
3. **Container Environment Variables** — Add > Name: `PROTON_BRIDGE_SMTP_PORT` / Value: `1026`
4. **Port Forwarding** — Add > Container Port: `25` / Node Port: `12025` (or any free port) / Protocol: `TCP`
5. **Storage** — Volumes > Mount Path: `/root` / Dataset name: `protonmail`
6. **Resource limits** — Check `Enable resource limits` and configure to your liking.

About step 3: on Kubernetes (used by TrueNAS Scale for Applications) the Bridge tends to listen on `127.0.0.1:1026` instead of `1025`. To confirm, open a console on the running pod and look for the `bridge` process in `netstat -ltpn`.

If everything is set correctly, on a TrueNAS Scale console:

```bash
sudo k3s kubectl get service --all-namespaces
```

should report the Bridge as:

```text
ix-protonmail-bridge    protonmail-bridge-ix-chart      NodePort    172.17.22.33    <none>  25:12025/TCP        1h
```

The SMTP server is then reachable on TCP port 12025 on your server's LAN IP address.

## Development

### Automated builds

A GitHub Actions workflow builds and pushes the images to both ghcr.io and Docker Hub. It:

1. Runs daily, and on pushes to `master` that touch the build files (manual runs can pin a version)
2. Fetches the latest Proton Mail Bridge version from the GitHub API and skips the build if that version is already published on ghcr.io
3. Builds both flavours for `linux/amd64` and `linux/arm64`
4. Tags them: Debian as `latest`, `debian`, `<version>`, `<version>-debian`; Alpine as `alpine`, `<version>-alpine`
5. Pushes to ghcr.io and Docker Hub (Docker Hub credentials come from GitHub secrets)
6. Prepends the new version to this changelog
7. Syncs this README to the Docker Hub repository description (README-only edits are synced by a separate lightweight workflow without rebuilding the images)

### Manual builds

With the helper script:

```bash
./build.sh              # build both flavours with the latest upstream version
./build.sh -v v3.25.0   # pin a specific version
./build.sh -p           # build and push to ghcr.io without prompting
```

Or directly with docker:

```bash
git clone https://github.com/asyncura/ProtonMailBridgeDocker.git
cd ProtonMailBridgeDocker/

# Debian flavour (default):
docker build --build-arg ENV_PROTONMAIL_BRIDGE_VERSION=v3.25.0 --tag=ghcr.io/asyncura/proton-mail-bridge .
# Alpine flavour:
docker build --file Dockerfile.alpine --build-arg ENV_PROTONMAIL_BRIDGE_VERSION=v3.25.0 --tag=ghcr.io/asyncura/proton-mail-bridge:alpine .
```
